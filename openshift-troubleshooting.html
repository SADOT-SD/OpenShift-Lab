Installation Troubleshooting Log

A chronological record of all errors encountered during the AWS IPI deployment and their architect-level resolutions.

Error 1: Installer Hangs Indefinitely at "INFO Credentials loaded..."

Symptom: The create install-config wizard froze immediately after loading AWS credentials and failed to prompt for the platform selection.

Root Cause:

The ~/.aws/credentials file contained an expired temporary session token (aws_session_token). The AWS API requests timed out because the temporary credentials were no longer valid.

Resolution:

Edited the ~/.aws/credentials file using nano and manually deleted the expired session token line.

Error 2: Incompatible Credentials Mode (Mint vs Passthrough)

Symptom: During create cluster, the installer threw a FATAL error: AWS credentials... are not valid for default credentials mode (and later failed for Passthrough mode as well).

Root Cause:

OpenShift strictly prohibits the use of temporary AWS credentials (keys with session tokens) for automated deployments. Temporary keys expire within hours, which would cause the cluster to permanently lose its ability to manage storage and scale nodes.

Resolution:

Created a dedicated IAM User in the AWS Console with the AdministratorAccess policy. Generated Long-Term Credentials (Access Key ID and Secret Access Key only, without a session token) and updated the Linux VM's configuration.

Error 3: AWS CLI "Unable to parse config file"

Symptom: Running aws configure resulted in a parsing error for the credentials file.

Root Cause:

Manual edits to ~/.aws/credentials corrupted the strict INI file formatting (likely deleting the [default] header).

Resolution:

Moved the corrupted file out of the way (mv ~/.aws/credentials ~/.aws/credentials.broken) and re-ran aws configure to cleanly generate a properly formatted file from scratch.

Error 4: Network Timeout & AddressLimitExceeded

Symptom: The installer timed out after 15 minutes during the network phase. Logs showed: api error AddressLimitExceeded: The maximum number of addresses has been reached.

Root Cause:

The installer attempts to build a highly available Multi-AZ architecture, requesting 3 Elastic IPs for 3 NAT Gateways. The AWS account had a default Service Quota limit of 5 Elastic IPs per region, and existing allocations pushed the request over the limit.

Resolution:

1. Executed ./openshift-install destroy cluster to clean up the partially built VPC.
2. Accessed the AWS EC2 Console and released unused Elastic IPs to ensure at least 3 were available.
3. Wiped the ocp-install directory, restored the clean install-config.yaml, and restarted the build.

Error 5: OpenShift CLI GLIBC Version Mismatch

Symptom: Running oc get nodes on the Linux VM returned: version 'GLIBC_2.34' not found (required by oc).

Root Cause:

The latest oc Linux binary was compiled against a newer version of the GNU C Library than what was installed on the Amazon Linux 2 Bastion host.

Resolution:

Pivoted to a local administration model. Copied the generated kubeconfig file from the Linux VM to a local Windows workstation, downloaded the native Windows oc.exe, and set the local KUBECONFIG environment variable.
